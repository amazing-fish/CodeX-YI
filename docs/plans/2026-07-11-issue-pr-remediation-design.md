# CodeX-YI 问题治理与分层 PR 设计

日期：2026-07-11

## 目标

- 明确项目当前问题的可复现症状、根因边界与验收标准。
- 将问题拆成可独立领取、可验证、可关闭的 GitHub Issues。
- 把已完成或可在当前周期闭环的修复组织为四个依赖清晰、便于评审的 PR。
- 补充 MIT 许可证，明确 `jiao-ling` 与贡献者的授权边界。

## 约束

- 继续使用 HTML、CSS、JavaScript，不引入框架、打包器或生产依赖。
- 数据加载必须失败关闭；空、缺失、畸形数据不得被标记为 ready。
- 历史记录属于不可信持久化输入，进入 DOM 前必须完成结构验证并使用安全渲染。
- 不覆盖原工作区的未提交改动；所有提交在隔离工作树和 `codex/` 分支完成。
- 每层 PR 都必须有独立验证证据，并同步维护 `ANCHOR.md`。

## PR 分层

### PR 1：许可与治理基线

基于 `main`。加入 MIT 许可证、设计文档与执行计划，记录审计锚点。它只改变授权和协作资产，不改变运行时行为。

### PR 2：静态界面与交互稳定性

基于 PR 1。收纳不依赖数据契约重构的安全修复：本地 favicon/字体、搜索 DOM 所有权、搜索序号与分类、分析器就绪判断、重复错误监听、投掷重置取消、必要的焦点恢复和回归测试。

### PR 3：数据加载与关系索引契约

基于 PR 2。将“收集数据”和“计算关系”拆为两遍，保证 64 个卦象各自拥有三类有效关系；对 64/8 数据规模、二进制编码、六爻结构和空/畸形输入执行失败关闭；补充契约测试。

### PR 4：历史记录安全边界

基于 PR 3。为 localStorage 历史建立显式 schema/codec；拒绝或迁移不合法旧记录；使用 DOM 文本节点渲染不可信内容；区分可保存的卦象详情与帮助/八卦等展示模型；加入持久化 DOM XSS 回归测试。

## 关键不变量

1. 应用只有在 64 个卦象和 8 个八卦均通过结构校验后才能进入数据就绪状态。
2. 关系索引必须在全量索引建立后计算，任何卦象都必须产生互、综、错三类有效引用。
3. localStorage 内容不能直接进入 `innerHTML`；历史记录只接受允许字段和合法值域。
4. 重置后，重置前启动的异步投掷不得继续更新状态或 UI。
5. 每个 PR 的代码、测试、锚点记录和 GitHub Issue 状态必须相互一致。

## 验证策略

- 每个 bug 先添加能复现失败的最小测试，再实现修复。
- 先运行对应 Node 契约测试，再运行全量测试和 JavaScript 语法检查。
- 数据变更额外验证 64/8 数量、编码唯一性、行结构与关系引用。
- 安全变更额外使用恶意 localStorage fixture 验证脚本不会执行。
- PR 创建后复核 base/head、远端 CI 与 Issue 关联，不自动合并。

## 非目标

- 本轮不引入 PWA、前端框架、构建系统或后端服务。
- 不在一个 PR 中一次性完成移动端、完整无障碍与 CI 治理。
- 不通过放宽数据校验或转义部分字符串来掩盖契约问题。
